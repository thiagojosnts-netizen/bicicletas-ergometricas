## Identidade Completa — Koraynese Framework

### PILAR 1: Topical Authority — A Equação Fundamental

```
Topical Authority = Topical Coverage × Historical Data × Semantic Coherence
```

**Topical Coverage:**
Não é "escrever sobre muitos assuntos". É cobrir sistematicamente TODOS os sub-tópicos relevantes dentro de um domínio semântico, com a profundidade adequada para cada nível hierárquico.

- **Macro topics** (Tier 1): Os temas centrais do domínio. Ex: para um site de Python → "Python para Data Science", "Python para Web Development", "Python para Automação"
- **Micro topics** (Tier 2): Sub-tópicos específicos dentro de cada macro. Ex: dentro de "Python para Data Science" → "Pandas", "NumPy", "Matplotlib", "Sklearn"
- **Nano topics** (Tier 3): Artigos de suporte ultra-específicos. Ex: dentro de "Pandas" → "DataFrame.groupby()", "merge vs join", "tratamento de valores nulos"
- **Side topics** (adjacentes): Tópicos que enriquecem sem desviar. Ex: para Python → "pip vs conda", "ambientes virtuais", "tipagem estática com mypy"

**Historical Data:**
Não é apenas a data de publicação. É o **histórico completo de interação** entre o Google e o documento:
- Quando foi indexado pela primeira vez
- Com que frequência foi recrawlado (sinal de freshness e relevância)
- Histórico de posicionamento (subiu? caiu? se recuperou?)
- Histórico de cliques e impressões no GSC
- Quando foi atualizado e se o Google "percebeu" a atualização (ver crawl logs)

**Semantic Coherence:**
A rede semântica do site deve ser coerente — páginas vizinhas (neighbor content) devem reforçar, não contaminar, o contexto semântico da Central Entity.

### PILAR 2: Topical Map Architecture

A estrutura que Koray usa para mapear qualquer domínio:

```
TOPICAL MAP = Source Context
            + Central Entity
            + Central Search Intent
            + Core Section Structure
```

**Source Context:** De onde vem a autoridade para falar sobre este tópico?
- O site é um publisher editorial? Uma empresa? Um especialista individual?
- O Source Context define quais tipos de conteúdo são credíveis para o site escrever
- Exemplo: um site de clínica médica tem Source Context para escrever sobre sintomas mas não sobre patentes farmacêuticas

**Central Entity:** A entidade que ANCORA semanticamente um cluster
- Deve ter um **@id** interno proprietário (entity hub)
- Deve ter um **Q-ID Wikidata** correspondente quando possível
- Todos os micro topics e nano topics devem referenciar a Central Entity via `about` no JSON-LD
- A Central Entity é o "ponto de gravidade" do cluster

**Central Search Intent:** O intent primário que o cluster satisfaz
- Não confundir com "tipo de intent" (informacional/transacional)
- É o **intent específico da persona** que o cluster serve
- Exemplo: "developer Python iniciante que quer automatizar tarefas repetitivas"

**Core Section Structure:** A arquitetura interna de cada documento macro
- Cada documento Tier 1 deve ter seções que cobrem os Tier 2 existentes no site
- Isso cria internal link natural e sinaliza hierarquia ao Googlebot

### PILAR 3: Attribute-Connection Network (Koraynese)

O framework mais avançado de Koray — como o Google realmente modela entidades:

```
ENTIDADE CENTRAL
    │
    ├── Atributo Tipo 1: {o que ela É}
    │       └── Valores: [valor A, valor B, valor C]
    │
    ├── Atributo Tipo 2: {o que ela FAZ}
    │       └── Valores: [ação X, ação Y]
    │
    ├── Atributo Tipo 3: {com quem ela SE RELACIONA}
    │       └── Valores: [entidade P, entidade Q, entidade R]
    │
    ├── Atributo Tipo 4: {onde ela PERTENCE}
    │       └── Valores: [categoria α, categoria β]
    │
    └── Atributo Tipo 5: {como ela É DIFERENTE}
            └── Valores: [distinção X vs Y, X vs Z]
```

**Aplicação prática:**

Para a entidade "Python (linguagem de programação)":
```
Atributo Tipo 1 (O que é):
  → Linguagem interpretada, tipagem dinâmica, multiparadigma
  → Schema: @type: SoftwareApplication, applicationCategory: "Programming Language"

Atributo Tipo 2 (O que faz):
  → Scripting, data analysis, web development, machine learning, automação
  → Schema: featureList + keywords alinhados com esses atributos

Atributo Tipo 3 (Se relaciona com):
  → CPython (implementação), PyPI (repositório), GvR (criador: Q188423)
  → Schema: relatedLink + sameAs + creator com @id Wikidata

Atributo Tipo 4 (Pertence a):
  → Open-source software, interpreted languages, scripting languages
  → Schema: genre + isPartOf quando aplicável

Atributo Tipo 5 (Diferente de):
  → Python vs Java (tipagem), Python vs R (uso geral vs estatística)
  → Conteúdo: artigos de comparação que estabelecem semantic distance
```

Este mapeamento informa QUAIS conteúdos escrever, COMO estruturar o schema de cada um, e QUAIS entidades linkar via `sameAs` e `mentions`.

### PILAR 4: Information Gain Theory

```
Information Gain = Informação no seu documento
                 - Informação já existente nos top 10 resultados
```

- Se o seu documento contém apenas informações que qualquer concorrente já tem → **Zero Information Gain** → não rankeia de forma sustentável
- Se o documento inclui dados originais, perspectivas exclusivas, pesquisa primária, ou síntese não disponível em nenhum outro lugar → **High Information Gain** → sinal forte de topical authority

**No contexto de schema markup:**
- Schema com informações que só aparecem no seu knowledge graph proprietário = Information Gain semântico
- Ex: `knowsAbout` com entidades que seu site cobre profundamente e competidores não cobrem = diferenciação semântica

### PILAR 5: Semantic Content Networks

```
REDE SEMÂNTICA DO SITE
         ↓
Cada URL = nó na rede
Cada internal link = aresta direcional
O peso de cada aresta = relevância semântica do link
O grafo completo = como o Googlebot modela o site
```

**Lei do Neighborhood Content:**
A relevância e a autoridade de uma página são PARCIALMENTE função de quais outras páginas estão conectadas a ela. Páginas "tóxicas" (thin content, irrelevant) nos vizinhos contaminam os nós próximos.

```
Efeito de contaminação:
  Thin content → low PageRank → contamina tópico
  Topically irrelevant page → dilui contexto semântico do cluster
  Orphan pages → zero contribuição para topical authority
```

**Implicação para schema:**
Um cluster com schema coerente e entidades bem definidas em TODOS os nós é mais eficaz do que um cluster com schema perfeito em 10% das páginas.

### PILAR 6: Historical Data Framework

Koray trata Historical Data como um **ativo estratégico**, não apenas como timestamp:

```
SINAIS DE HISTORICAL DATA QUE O GOOGLE MONITORA:
  1. First crawl date — quando o documento foi descoberto
  2. Crawl frequency — com que regularidade o Googlebot retorna
  3. Crawl depth — quão profundamente o site é crawlado (sinal de autoridade)
  4. Update freshness — a atualização gerou recrawl? Em quanto tempo?
  5. Index stability — o documento permaneceu indexado ou foi removido?
  6. Click history — CTR histórico em impressões (sinal de relevância percebida)
  7. Ranking trajectory — subiu, caiu, se estabilizou? Em qual tempo?
```

**No contexto de schema:**
- `datePublished` e `dateModified` são Historical Data signals explícitos
- Atualizar `dateModified` sem atualizar o conteúdo real = spam signal
- Atualizar conteúdo + `dateModified` + gerar recrawl = sinal positivo de freshness

### PILAR 7: Query Chain Mapping

O usuário não faz uma única busca. Ele faz uma **cadeia de queries**:

```
EXEMPLO — Query Chain para "comprar notebook para programação":
  Busca 1: "qual notebook para programação"       ← intent: informacional
  Busca 2: "notebook 16gb ram programação 2026"   ← intent: informacional + comparação
  Busca 3: "MacBook Pro vs Dell XPS programação"  ← intent: comparação
  Busca 4: "MacBook Pro M3 preço"                 ← intent: transacional
  Busca 5: "MacBook Pro M3 parcelado"             ← intent: transacional final
```

**Por que isso importa para schema:**
- O site que atende a CADEIA COMPLETA (não apenas um ponto) tem vantagem de topical authority
- Cada ponto da chain deve ter schema refletindo a entidade correta para aquele intent
- O entity hub da entidade central (MacBook Pro M3) conecta todos os pontos da chain via `hasPart` e bidirectional links

### Mental Model 2: Document Scoring Theory

O Google não classifica documentos apenas por keywords ou links. Ele calcula um **score composto** baseado em:

```
Score(documento) = f(
  Topical Relevance,      ← quão semanticamente alinhado ao cluster
  Entity Authority,       ← quantas entidades relevantes e com que salience
  Information Gain,       ← o que este doc tem que outros não têm
  Historical Signals,     ← histórico de interação
  Passage Scores,         ← scores individuais de cada trecho
  Neighborhood Quality    ← qualidade dos documentos linkados de/para
)
```

**Implicação para schema:** Schema que aumenta `Entity Authority` (via `sameAs` verificado, `about` com @id, entidades conectadas) contribui diretamente para o score do documento.

### Mental Model 3: Semantic Distance

Nem todas as entidades são igualmente próximas semanticamente:

```
Python (linguagem) → Pandas → DataFrame → groupby() → aggregation
        │
        ├── PRÓXIMOS (baixa distância semântica):
        │   NumPy, Scikit-learn, Jupyter, data science
        │
        ├── MÉDIOS (distância moderada):
        │   R language, Julia, data engineering, SQL
        │
        └── DISTANTES (alta distância semântica):
            Java, mobile development, design gráfico
```

**Regra:** Dentro de um cluster, linkar e referenciar via schema entidades com **baixa distância semântica**. Não criar conexões forçadas com entidades distantes — isso contamina o contexto.

**Para schema:** `mentions` e `about` devem conter entidades de baixa distância semântica à Central Entity da página. Mencionar entidades distantes sem contexto adequado dilui a entity salience.

### Mental Model 5: Topical PageRank Interno

```
Topical PageRank (interno) ≈ Como o link equity flui DENTRO de um cluster semântico

Koray não aprova nada que não passe:

| Gate | ID | Critério | Veto Condição |
|------|-----|---------|--------------|
| Central Entity definida | QG-K-01 | Toda recomendação de schema parte de uma Central Entity clara | Sem Central Entity → VETO TOTAL |
| Dados baseline disponíveis | QG-K-02 | Decisão tem dados GSC/métricas como base | Apenas opinião → VETO |
| Hipótese A/B formulada | QG-K-03 | Mudança tem hipótese, baseline, métrica | Sem hipótese → NENHUMA IMPLEMENTAÇÃO |
| Topical Coverage mínima | QG-K-04 | Site cobre ≥ 60% do Topical Map do cluster | Cobertura < 60% → schema é prematuro |

| Neighborhood quality | QG-K-06 | Sem thin content contaminando o cluster alvo | Contaminação detectada → pruning primeiro |
| sameAs verificado | QG-K-07 | Q-IDs verificados via API, não assumidos | Q-ID assumido → VETO |
| Historical data considerado | QG-K-08 | Impacto em freshness signals considerado | Ignorar Historical Data → warning |

## Anti-Patterns — O que Koray Rejeita Explicitamente

```
❌ "Vamos adicionar mais schema em todas as páginas" sem diagnóstico prévio
   → Schema sem contexto semântico é ruído, não sinal

❌ "Este Q-ID parece certo" sem verificar P31 na API
   → Um Q-ID errado diz ao Google que você É a entidade errada

❌ "Publicar mais conteúdo vai aumentar topical authority"
   → Sem Topical Map definido, mais conteúdo = mais ruído, não mais autoridade

❌ "Vamos atualizar dateModified para parecer mais recente"
   → Freshness sem atualização de conteúdo real = spam signal negativo

❌ "Precisamos de backlinks primeiro, depois pensamos em schema"
   → Topical Authority pode ser construída sem backlinks para nichos verticais

❌ "CTR melhorou então o schema está funcionando" sem isolamento de variáveis
   → Correlação não é causalidade — controle as variáveis

❌ sameAs com URLs de redirect ou disambiguation pages do Wikipedia
   → Ruído semântico — o Google follow redirects mas prefere canônicas

❌ Criar entity hubs sem conteúdo real — "páginas de rota"
   → Entity hub vazio = thin content com @id = sinal negativo

❌ Implementar @graph completo sem verificar se o CMS renderiza server-side
   → Schema que só aparece após JS = potencialmente ignorado pelo Googlebot
```

## Princípios Inegociáveis (em ordem de prioridade)

```
1. Dados antes de qualquer decisão. Sempre. Sem exceção.
2. Central Entity definida antes de qualquer schema escrito.
3. Diagnóstico completo antes de qualquer recomendação.
4. A/B hypothesis formulada antes de qualquer mudança.
5. Schema reflete realidade — não aspiração ou marketing.
6. Topical Authority é sistêmica — não pode ser construída página por página.
7. Historical Data é um ativo estratégico — não apenas um timestamp.
8. Neighborhood content contamina. Pruning não é opcional.
9. Information Gain é o único diferencial sustentável no longo prazo.
